# Authentication Documentation Realignment

> **Status — 2026-04-21:** Superseded by [`2026-04-21-auth-agnostic-posture-design.md`](2026-04-21-auth-agnostic-posture-design.md), which moves the project from informational-neutral path-documentation to full auth-agnosticism. This spec is preserved as a historical record.

**Status:** Draft
**Date:** 2026-04-17
**Author:** jnurre64 (with Claude)

## Problem

The project's documentation takes a strict "Console API key only" posture for authenticating the `claude` CLI on a self-hosted runner. This posture is expressed across `README.md`, `docs/security.md`, `docs/runners.md`, `docs/faq.md`, `.claude/skills/setup/SKILL.md`, `docs/getting-started.md`, and `scripts/setup.sh`, and explicitly warns users **against** substituting `CLAUDE_CODE_OAUTH_TOKEN`.

Two problems with the current stance:

1. **It's over-restrictive for an individual solo developer.** Anthropic's official Claude Code authentication docs explicitly document `claude setup-token` → `CLAUDE_CODE_OAUTH_TOKEN` as the supported credential type for "CI pipelines and scripts where browser login isn't available." Consumer Terms carve out this exact scenario via their "except where we otherwise explicitly permit it" clause. For a solo developer using `claude -p` on their own self-hosted runner against their own repo, OAuth is within the spirit and letter of the ToS.

2. **It's quietly inaccurate in the other direction.** The README's "No additional subscriptions" feature bullet implies the system is free-to-run for everyone, but that framing only holds for individual subscription users. Teams, shared-runner deployments, or anything with commercial characteristics require an Anthropic Console API key (billed separately), which the README does not acknowledge.

The goal of this work is to **inform users, not prescribe a choice.** The repository should present both paths as legitimate, document what makes each path ToS-compliant (and what would violate it), then step out of the way. Out-of-the-box, the repo remains ToS-compliant under either path. Responsibility for misuse stays with the user.

## Non-goals

- **Not** adding a runtime branch in the dispatch scripts. `scripts/lib/common.sh` already invokes `claude -p` without specifying which env var is set; Claude Code's own authentication precedence picks up whichever is present. No code change is needed.
- **Not** offering an interactive auth-choice prompt in `scripts/setup.sh`. Adding a branched prompt increases surface area for bugs and gives the shell installer too much opinion about auth. The `/setup` skill is a better place for branching.
- **Not** documenting `ANTHROPIC_AUTH_TOKEN` (the Bearer-token path for LLM gateways). Out of scope for this project; only relevant to enterprise gateway deployments that aren't the target audience.
- **Not** addressing `--bare` mode. This repo does not use `--bare` (verified at `scripts/lib/common.sh:155-156`), and adding documentation for a mode we don't use invites drift.
- **Not** re-framing the tool as a consumer or commercial product. The repo remains a development toolkit; the authentication discussion is purely about how the user wires up their Anthropic credentials on the runner.

## Ground truth (verified against Anthropic sources on 2026-04-17)

The spec treats these as the authoritative facts. Citations are embedded so future reviewers can re-verify.

### What is explicitly supported for `claude -p`

Anthropic's Claude Code authentication precedence order (from `code.claude.com/docs/en/authentication`):

1. Cloud provider credentials (Bedrock/Vertex/Foundry)
2. `ANTHROPIC_AUTH_TOKEN` (Bearer-token gateway path; out of scope)
3. `ANTHROPIC_API_KEY` (Console API key — direct API access)
4. `apiKeyHelper` (dynamic credential script; out of scope)
5. `CLAUDE_CODE_OAUTH_TOKEN` (long-lived OAuth from `claude setup-token`, 1-year TTL, subscription-backed)
6. Subscription OAuth from `/login` (browser flow)

Verbatim from the docs on `CLAUDE_CODE_OAUTH_TOKEN`:

> "A long-lived OAuth token generated by `claude setup-token`. Use this for CI pipelines and scripts where browser login isn't available."
>
> "This token authenticates with your Claude subscription and requires a Pro, Max, Team, or Enterprise plan. It is scoped to inference only and cannot establish Remote Control sessions."

This is the explicit permission that satisfies Consumer Terms Section 3's "automated access except where we otherwise explicitly permit it" carve-out.

### What the April 4, 2026 policy update changed

Anthropic tightened restrictions on **third-party tools** (OpenClaw, Cline, Cursor, etc.) routing requests through subscription OAuth, moving from outright ban (Feb 2026) to paid usage bundles (April 2026). Per Boris Cherny, Head of Claude Code at Anthropic:

> "Claude Code is unaffected, and your subscription quota still fully covers its usage."

This is critical for our context: this repo invokes the native `claude` binary via `claude -p`, not the Agent SDK and not a third-party harness. It sits squarely in the "Claude Code is unaffected" bucket.

Distinguishing this from `anthropics/claude-code-action` matters: that action's v1.0 README now lists only "Anthropic direct API, Amazon Bedrock, Google Vertex AI, Microsoft Foundry" as supported auth — OAuth is no longer emphasized there. But we are not using that action. We invoke the native CLI directly, which preserves the stronger ToS posture.

### Absolute prohibitions (apply regardless of path)

From Anthropic's `legal-and-compliance` page and Consumer Terms, verbatim or paraphrased:

- Subscription OAuth may **not** be used with the Agent SDK (Python/TypeScript) — those are products built on Claude, not native Anthropic applications
- Subscription OAuth may **not** be used by third-party tools to route requests on behalf of their end users
- "You may not share your Account login information, Anthropic API key, or Account credentials with anyone else" — routing multiple humans through one OAuth token violates this
- Subscription usage limits "assume ordinary, individual usage of Claude Code" — sustained high-volume automation or 24/7 services pushes outside this boundary

### The silent-override footgun

When both `ANTHROPIC_API_KEY` and `CLAUDE_CODE_OAUTH_TOKEN` are present in the environment, Claude Code silently prefers the API key (item #3 before item #5 in precedence). A user who thinks they're on the OAuth path will be silently billed to their Console account. The documented verification is `claude /status`.

## Design

The design is informational-neutral. The docs present both paths, document all ToS-bounding guardrails, and leave the choice to the user.

### New file: `docs/authentication.md`

Single source of truth for auth. Every other doc links here rather than duplicating nuance.

**Structure (in this order — safety first):**

1. **Overview** — one paragraph: two supported paths (API key and OAuth token), both invoke the same `claude -p` binary, the scripts are agnostic to which one is set.

2. **Terms of Service boundaries** — *first section after overview, before the decision matrix.* A short list of the absolute rules that apply to either path:
   - Never share a subscription OAuth token with another human (direct Consumer Terms violation)
   - Never set the token as a secret on a shared-access runner where multiple developers' workflows use it
   - Never forward OAuth tokens to the Agent SDK, third-party tools, or custom harnesses — only the native `claude` CLI
   - Subscription usage limits assume "ordinary, individual usage" — sustained high-volume automation violates this
   - If both `ANTHROPIC_API_KEY` and `CLAUDE_CODE_OAUTH_TOKEN` are set, API key silently wins — verify with `claude /status` after setup

3. **Decision matrix** — a table mapping use context to auth path:

   | Your situation | Recommended auth | Why |
   |---|---|---|
   | Solo developer, your own repo, your own self-hosted runner | Either works — pick based on your preference and billing model | Both paths are documented by Anthropic |
   | Team with shared runner(s), multiple developers trigger issues | `ANTHROPIC_API_KEY` (Console API key) required | OAuth would share credentials across humans |
   | Commercial product, customer-facing service, 24/7 operation | `ANTHROPIC_API_KEY` required | Subscription terms limit to "ordinary individual usage" |
   | Any integration with the Agent SDK or third-party tools | `ANTHROPIC_API_KEY` required | Anthropic explicitly prohibits OAuth for these |

4. **Path A: Console API key (`ANTHROPIC_API_KEY`)** — when to use, how to get one, how to set it in the runner's `.env`. Includes explicit note: "This is the required path for team, commercial, or shared-runner deployments."

5. **Path B: Subscription OAuth token (`CLAUDE_CODE_OAUTH_TOKEN`)** — when it's appropriate (individual use against your own repo), how to generate with `claude setup-token`, how to set it in the runner's `.env`. Explicit preconditions:
   - You have a personal Pro, Max, Team, or Enterprise subscription in good standing
   - The runner is effectively your machine (not shared with other humans)
   - The repo is yours (or one where you're the only human adding the `agent` label)
   - You understand the token expires after ~1 year and must be regenerated

6. **Never-OK patterns** — a short list of the misuse patterns that would violate ToS under either path. Reuses items from the Boundaries section but in "anti-pattern" framing with concrete examples.

7. **Verification checklist** — what to run after configuring either path:
   - `claude /status` shows the auth method you expect
   - `echo $ANTHROPIC_API_KEY` and `echo $CLAUDE_CODE_OAUTH_TOKEN` — exactly one should be set
   - Test run completes without auth errors; `claude-stderr-*.log` is clean

8. **Operator responsibilities** — short, scannable:
   - Rotate Console API keys every 90 days
   - Regenerate OAuth tokens before 1-year expiry
   - Ensure `.env` is `chmod 600`
   - Don't commit `.env` or credentials to the repo
   - Confirm the key/token belongs to an account/workspace appropriate to your use (personal vs. organization; Consumer vs. Commercial Terms)

9. **References** — direct links to:
   - `code.claude.com/docs/en/authentication`
   - `code.claude.com/docs/en/legal-and-compliance`
   - `www.anthropic.com/legal/consumer-terms`
   - `www.anthropic.com/legal/commercial-terms`
   - `www.anthropic.com/legal/aup`
   - `support.claude.com/en/articles/11145838-using-claude-code-with-your-pro-or-max-plan`

   The April 4, 2026 policy update is mentioned in the body text as context. Do not require a citation to a specific secondary-source article (Hacker News thread, trade press) — the load-bearing citation is the Claude Code authentication doc itself, which continues to document `CLAUDE_CODE_OAUTH_TOKEN` as the supported credential for CI after the April 4 update. Reference Anthropic's own docs rather than third-party coverage.

10. **Disclaimer** — retain the existing "not legal advice" caveat from `security.md`. Users should review Anthropic's current terms themselves.

## Risks and trade-offs

**Risk: Doc drift if Anthropic changes policy again.** The April 4, 2026 update is less than two weeks old. Another tightening is possible. Mitigation: `authentication.md` cites Anthropic's primary sources; when their policy changes, the update to our docs is localized to one file. The other docs just link to `authentication.md`.

**Risk: Users pick OAuth for team deployments despite the guardrails.** The neutral framing explicitly does not steer users. Mitigation: the "Terms of Service boundaries" section is placed before the decision matrix, and the decision matrix makes "team" → API key unambiguous. Beyond that, responsibility for choosing the correct path is the user's.

**Risk: The "ordinary individual usage" language has no bright-line threshold.** Anthropic has not defined a request-per-day or runs-per-week cap. Mitigation: frame the boundary qualitatively (24/7 service patterns, extreme volume, multi-user routing) and recommend API key for anyone near the edge. When in doubt, the user can contact Anthropic Sales — link provided.

**Trade-off: The `/setup` skill becomes slightly more complex.** Adding a branch in Step 9d adds a decision point that wasn't there before. Mitigation: the branch is small (3-4 extra lines), and `authentication.md` carries the decision logic — the skill just asks the question and routes.

**Trade-off: Some users may prefer stronger guidance.** A user who wants "just tell me what to do" gets a decision matrix instead of a default. Mitigation: for those users, the matrix's first row ("individual solo developer") names both paths as acceptable, giving them license to pick either without overthinking. The checklist at the bottom is directive.

## Implementation order

When this design moves to an implementation plan, the natural order is:

1. Write `docs/authentication.md` first (everything else links here)
2. Update `docs/security.md`, `docs/runners.md`, `docs/faq.md`, `docs/getting-started.md`, `README.md` in any order (all link to authentication.md)
3. Update `.claude/skills/setup/SKILL.md`
4. Update `scripts/setup.sh` (one-line link addition)
5. Verification: read through the doc set end-to-end and confirm internal links resolve and messaging is consistent

## Out of scope for a follow-up

These are noted for posterity but not part of this spec:

- **Local Docker-based runner guide.** The `claude-work/` reference document includes a whole section on Docker sandbox patterns. The user explicitly deferred this.
- **Subscription lifecycle automation.** Detecting when an OAuth token is about to expire and warning the operator in agent logs could be future work but is not part of this doc update.
- **Usage-pattern guardrails in the circuit breaker.** The existing circuit breaker limits per-issue comment volume; a separate design could add cross-issue rate-limiting for users on the OAuth path, but is outside this spec.
