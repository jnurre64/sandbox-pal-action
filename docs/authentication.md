# Authentication

Claude Agent Dispatch runs `claude -p` (headless mode of the native Claude Code CLI) on a self-hosted GitHub Actions runner. Claude Code supports multiple authentication methods; this project is agnostic to which is configured — Claude Code's own authentication precedence handles the rest.

Two paths are relevant to this project:

- **`ANTHROPIC_API_KEY`** — a Console API key (`sk-ant-api...`) issued from the [Anthropic Console](https://console.anthropic.com). Billed per token against your Console account.
- **`CLAUDE_CODE_OAUTH_TOKEN`** — a long-lived OAuth token (`sk-ant-oat01-...`) generated by [`claude setup-token`](https://code.claude.com/docs/en/authentication). Backed by your Pro/Max/Team/Enterprise subscription.

Both invoke the same native `claude` binary. This page lays out the Terms of Service boundaries common to both, a decision matrix for picking between them, and verification steps.

## Terms of Service boundaries

These apply regardless of which path you choose. They are the rules that make each path ToS-compliant.

- **Never share a subscription OAuth token with another human.** Anthropic's Consumer Terms explicitly prohibit sharing account credentials: *"You may not share your Account login information, Anthropic API key, or Account credentials with anyone else."* Routing multiple developers through one OAuth token is a direct Consumer Terms violation.
- **Never set the token as a secret on a shared-access runner** where workflows triggered by other humans will consume your subscription quota. Individual use means *your* runner, triggered by *you*.
- **Never forward OAuth tokens to the Agent SDK, third-party tools, or custom harnesses.** Anthropic's [legal and compliance page](https://code.claude.com/docs/en/legal-and-compliance) is explicit: *"Developers building products or services that interact with Claude's capabilities, including those using the Agent SDK, should use API key authentication through Claude Console."* Only the native `claude` CLI may use subscription OAuth.
- **Subscription usage limits assume "ordinary, individual usage."** Per Anthropic's [legal and compliance page](https://code.claude.com/docs/en/legal-and-compliance): *"Advertised usage limits for Pro and Max plans assume ordinary, individual usage of Claude Code and the Agent SDK."* Sustained high-volume automation or 24/7 service patterns push outside this boundary.
- **Never set both env vars simultaneously.** `ANTHROPIC_API_KEY` takes precedence over `CLAUDE_CODE_OAUTH_TOKEN` in Claude Code's resolution order. If both are set, the Console API key is used silently and charges route to your Console account. Verify the active path with `claude /status` after configuring the runner.

## Decision matrix

| Your situation | Recommended auth | Why |
|---|---|---|
| Solo developer, your own repo, your own self-hosted runner | Either works — pick based on your billing model and preference | Both paths are documented by Anthropic |
| Team with shared runner(s), multiple developers trigger issues | `ANTHROPIC_API_KEY` required | OAuth would share credentials across humans — Consumer Terms violation |
| Commercial product, customer-facing service, or 24/7 operation | `ANTHROPIC_API_KEY` required | Subscription terms limit to "ordinary individual usage" |
| Any integration with the Agent SDK or third-party tooling | `ANTHROPIC_API_KEY` required | Anthropic explicitly prohibits OAuth for these |

## Path A: Console API key

**When:** Required for team deployments, shared-access runners, commercial/customer-facing use, high-volume or 24/7 operation, or any integration with the Agent SDK or third-party tools. Works equally well for solo use if you prefer per-token billing.

**How to configure:**

1. Generate an API key at [console.anthropic.com](https://console.anthropic.com). The key will begin `sk-ant-api...`.
2. On the runner machine, add the key to the runner's `.env` file (in the runner installation directory — not `~/.bashrc`; systemd services do not source shell profiles):

   ```bash
   echo 'ANTHROPIC_API_KEY=sk-ant-api...' >> .env
   chmod 600 .env
   ```

3. Restart the runner service to pick up the new environment.

**Billing:** Per-token, against the Anthropic account that owns the key.

## Path B: Subscription OAuth token

**When:** Acceptable for an individual solo developer running this project against their own repo on their own self-hosted runner, with a Pro/Max/Team/Enterprise subscription in good standing.

**Preconditions:**

- You have an active Pro, Max, Team, or Enterprise subscription
- The runner machine is effectively your machine — no other humans trigger workflows on it
- The repo is yours, or one where you are the only human adding the `agent` label and commenting on issues
- You accept that this usage counts against your subscription's "ordinary individual usage" quota

**How to configure:**

1. On a machine where you have already logged into Claude Code (`claude /login`), generate a long-lived token:

   ```bash
   claude setup-token
   ```

   The command walks you through OAuth authorization and prints a token that begins `sk-ant-oat01-...`. It is not stored anywhere — copy it immediately.

2. On the runner machine, add the token to the runner's `.env` file:

   ```bash
   echo 'CLAUDE_CODE_OAUTH_TOKEN=sk-ant-oat01-...' >> .env
   chmod 600 .env
   ```

3. Confirm `ANTHROPIC_API_KEY` is **not** set in the same `.env` (it would silently override the OAuth token).

4. Restart the runner service to pick up the new environment.

**Token lifecycle:** The token is valid for approximately one year, and only while your subscription remains active. Regenerate it before expiry or when rotating credentials.

**Billing:** Covered by your existing Pro/Max/Team/Enterprise subscription — no separate per-token charges.

## Never-OK patterns

These violate the Terms of Service regardless of which path you configured:

- **Setting `CLAUDE_CODE_OAUTH_TOKEN` on a runner that multiple humans can trigger workflows on.** Even a private team repo is a multi-human environment if more than one person can add the `agent` label. Use `ANTHROPIC_API_KEY` instead.
- **Committing either credential to any repository, including private ones.** Both belong in the runner's `.env` file, `chmod 600`.
- **Copying your `~/.claude/` credentials directory into containers, CI systems, or other developers' environments.** This is credential forwarding and falls outside the supported Claude Code authentication paths.
- **Using `CLAUDE_CODE_OAUTH_TOKEN` with the Agent SDK, OpenClaw, Cline, Cursor, or any harness other than the native `claude` CLI.** API key required.
- **Running this dispatch system as a service for customers or end users.** That is commercial use — Console API key under Commercial Terms, not subscription OAuth.

## Verification

After configuring the runner, verify the authentication is what you expect:

- [ ] `echo $ANTHROPIC_API_KEY` and `echo $CLAUDE_CODE_OAUTH_TOKEN` — exactly one should return a value; the other should be empty.
- [ ] Run `claude /status` on the runner (as the runner user, with the `.env` sourced). It should report the auth method you configured.
- [ ] Trigger a test run of the dispatch system against a benign issue. Confirm the run completes without auth errors and that `~/.claude/agent-logs/claude-stderr-*.log` is clean for that run.
- [ ] For the OAuth path only: confirm usage registers against your subscription by checking your plan's usage dashboard (not the Console).
- [ ] For the API key path only: confirm the key is active and the Console workspace is the one you intended to bill.

## Operator responsibilities

Regardless of path:

- `.env` file on the runner must be `chmod 600` — readable only by the runner user.
- Never commit `.env` or credentials to any repository.
- Confirm the account or workspace that owns the credential is appropriate for your intended use (personal vs. organization; Consumer vs. Commercial Terms).
- Monitor `~/.claude/agent-logs/` for authentication failures and stderr output.

For **Console API keys**: rotate every 90 days. Update the `.env` file with the new key and revoke the old one only after verifying the new key works.

For **subscription OAuth tokens**: regenerate before the ~1-year expiry. If you let the token lapse or cancel the underlying subscription, dispatch runs will fail with authentication errors.
